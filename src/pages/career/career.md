# 自己PR

職務経歴を貫く強みは、①エンジニアリングを土台にしたデータドリブンな意思決定、②AI を組織の集合知に変換する発想、③不確実性のオーガナイズ、④仕組みを考案し委譲するサイクル、の4つです。具体的な事例は職務経歴を参照ください。

## 1. エンジニアリングを土台にしたデータドリブンな意思決定

技術制約とユーザー価値の両方を理解した上で、グロースハック／マーケティング理論と数値分析を結びつけ、施策を発案から実行まで持っていけることが強みです。「分析できるエンジニア」「実装できる PdM」のどちらでもなく、両者を同じ思考プロセスの中で扱えることを重視しています。

意思決定の現実解は、ユーザー価値・技術制約・データから読める挙動の3つが交差する地点にあると考えており、そこに着地させる判断軸を持っていることが価値だと考えています。チーム内では分析リテラシー自体を底上げする教育にも投資しており、自分一人がボトルネックにならない構造づくりを意識しています。

## 2. AI を組織の集合知に変換する発想

Coding Agent の調教（プロンプト・コンテキスト・MCP 設定）を個人の生産性ハックに閉じさせず、リポジトリにコミットして組織で共有・進化させる対象として扱う、という思想を一貫して持っています。

AI 活用は個人スキルではなく組織能力として設計すべきだと考えており、要件定義・デザイン・コードレビューといった既存プロセスを「AI が介在することを前提にどう再設計すべきか」という観点で見直しています。シニアエンジニアの暗黙知を AI 経由で再利用可能な形に変換することにも関心があり、属人化していたレビュー知見を組織資産化する試みを継続しています。

## 3. 不確実性のオーガナイズ

研究駆動・0→1 領域のプロジェクトで「いつ・何を・どこまで出すか」のバランス調整を中心的な仕事として捉えています。学生時代からピクシブ R&D、WindHole の創業期、CoeFont の研究連携プロジェクトと、不確実性の高い領域に 7 年以上継続して身を置いてきました。

このタイプの仕事で重要なのは、研究側の成果と事業側の要請の間で「出せるのか／何を待てば出せるのか」を翻訳することと、関係者間の依存関係を可視化して合意形成のリードタイムを短くすること。ドキュメント設計（要件定義書・エンティティ別 CRUD／ライフサイクル／ビジネスロジック分割）と部門横断のコミュニケーション設計を方法論として持っています。

## 4. 仕組みを考案し委譲するサイクル

「このチームには XX が足りない」と気づいたら、制度を設計し運用に乗せ、定着したら後任に委譲する。このサイクルを回し続けることがマネージャー業務の核だと考えています。

新しい役割や運用を作って自分が背負い続けるのではなく、再現可能なフローに落とし込んで他のメンバーに渡すことを最初から設計に織り込みます。土台にあるのは、エンジニア・ステークホルダー間の橋渡しの実務経験です。技術的な専門用語を平易な例えに置き換えながら、経営層・ビジネスサイドとエンジニアの間のコミュニケーションを担ってきました。

#### 4. AI コーディングエージェントを前提とした開発プロセス再設計

2025年以降、Claude Code が実用に乗ったタイミングで開発プロセス全体を AI 前提に再設計することを主導。基本思想は、Coding Agent の調教（プロンプト・コンテキスト・MCP接続）を個人最適化に閉じさせず、リポジトリにコミットして組織の集合知として育てていくこと。

- **要件定義プロセスの GitHub 移行**：PRD・要件・PBI を Markdown で GitHub 管理。Notion で散在していた要件定義を Diff / Review / 履歴管理可能な状態にし、PR ベースで PRD レビューを受けるフローへ転換。
- **Claude Code への調査基盤接続**：調査系 MCP・社内 API を Claude Code に接続し、要件作成の調査・背景情報補足・PRD 妥当性評価を Agentic に回せる基盤を整備。
- **エージェント定義のリポジトリ化**：各メンバーのエージェント定義（プロンプト・カスタムコマンド・MCP 設定）をリポジトリにコミットする運用に統一。
- **レビュー知見の AI プロンプト化基盤**：GitHub PR のレビューコメントを diff hunk・ファイルパス・行番号・コミット情報とともに構造化 JSON で抽出する CLI ツール [pr-review-data-collector](https://github.com/krgpi/pr-review-data-collector) を自作。シニアエンジニアの過去レビュー指摘を AI コードレビュー Bot 用の学習・プロンプトデータセットとして再利用できる形に整え、暗黙知を組織横断で活用する仕組みを試作。

設立: 2024年（2023年11月よりプロジェクト稼働）  
従業員数: 4名（共同創業メンバー、全員兼業）  
役職: 共同創業者／プロダクト責任者

- [カギスル｜パスワード付き ZIP の代替、ファイル送受信の暗号化サービス](https://kagisuru.ideoj.co.jp/)
- [株式会社IDEOJ](https://ideoj.co.jp/)

### 概要

筑波大学の同期と共同で創業したソフトウェア開発の会社で代表を務めています。PPAP（パスワード付き ZIP メール送付）の代替を狙う BtoB SaaS プロダクト「カギスル」の開発・運営会社です。3か月単位のストラテジードキュメントを策定し、毎週の経営 MTG／隔週の開発 MTG で進捗確認・軌道修正するサイクルで経営しています。兼業4人体制という非常に少ないリソースの中で、プロダクト企画開発・マーケティング＆セールス・ファイナンスを横断して担当しています。

### 詳細

#### 1. プロダクト企画開発

##### 機能企画から PRD・実装・リリースまでの一貫プロセス整備

機能アイデアから本番リリースまでの過程で「**PRD（指定フォーマットによる要件定義・意思決定の透明化） → ドメインモデリング／ワイヤーフレーム**」の特に重要なステップでの意思決定プロセスをドキュメントテンプレートで標準化し、エンジニアとPMのコミュニケーションや共通認識の構築を追跡可能な形で残しながら進めています。

##### Phase 制によるロードマップ管理

クオーターを節目に、開発フェーズを **デスクトップアプリMVP 構築 → 販売準備〜MVP リリース → プレスリリース・ブランド刷新 → Webアプリ移行完結・展示会 → 法人受け入れ態勢 → 負債解消・流入導線最適化**と明示的に切り替えながら運用。各フェーズで「何を作るか」より「何を作らないか」を先に合意するスタイルで、兼業メンバーのリソースをエピック単位で集中させています。

##### 主要機能の企画・実装リード

- **ブラウザ完結の暗号化／復号 UX**：アプリ未インストールでも Web 経由で暗号化／復号できる体験を要件定義から実装まで担当。アクティベーション障壁を大幅に低減。
- **WebAuthn + PRF Extension の技術検証**：Web 完結で秘密鍵管理と暗号化を実現できるかをスパイク。「パスキー（FIDO2）によるファイル暗号化」という訴求軸の技術的裏付けを確立。
- **アクティベーション UX 再設計**：「未登録ユーザーへの送信時に、受取側のアカウント作成を必須から任意に切り替える」「暗号化処理であることをアニメーションで明示し、ユーザーが押すボタンの意味を一意化する」など、アクティベーション率向上に直結する変更を発案・優先実装。
- **組織アカウント機能**：法人顧客の受け入れ態勢として、組織アドレス・組織メンバー管理を実装し、対外発表のマイルストーンに間に合わせる形でリリース。
- **メールクライアント連携機能**：主要メーラーのアドインから、メール作成画面内で暗号化と添付を完結する UX を実現。受取人候補としてメールクライアントの送信履歴を活用することで入力負担を削減。

#### 2. マーケティング ＆ セールス

##### Product-Led Growth 設計：リファラル誘発を組み込んだユーザーグロースモデル

創業初期から「**宣伝費をかけない・リファラルで増やしていける回路を組む**」を費用方針として明文化。ターゲットユーザーがプロダクトを使ってファイルを送付した相手に、サービス自体が自動的に認知される構造をプロダクトの根幹に据え、暗号化ファイルを受け取った相手が復号フローを通じて自然にサービスに触れる導線を設計。アクティベーション → リファラル → ステップアップという順でユーザーフローを点検する方針を Sprint MTG で繰り返し合意し、流入の確保よりもまず「**取りこぼさない**」「**次のユーザーに広げる**」UX に集中する優先順位を維持しています。

##### 展示会出展

国内主要スタートアップ展示会・大手 IT メディア主催の展示会など複数回の出展を担当。申し込みから体験型展示内容の企画や、当日来場者ヒアリングから得た質問群（「鍵の登録・交換はどうするのか」「既存の PPAP 代替サービスと何が違うのか」「オンプレ／イントラネット需要」など）を即日サービスの修正方針として整理し、直後の経営 MTG にフィードバックして週次の方針アップデートに反映するサイクルを確立。デモブース出展に向けて、デモ環境（ステージング／ローカル）・看板・QR コード付き資料・トークスクリプトをパッケージとして準備するフローを定型化しました。

##### 公的支援機関の活用

中小企業向けの公的支援機関（無料マーケ支援／ビジネスマッチング）の活用を方針化。申請書類作成、訪問審査での競合優位性・市場性・収益可能性・体制・経営課題の質疑応答準備、複数年にわたるビジネスマッチング支援、公的媒体への出稿、発表会への出展まで一連の対応を完遂しました。

##### テレアポと商談運用（営業代行パートナー連携）

営業代行パートナーと契約し、月1000件規模のテレアポ運用を開始。並行して、自社で営業の品質を担保するためのトークスクリプト・ROI 提案資料・紹介資料を準備。特定業種（医療機関）向けに数百社規模のテストマーケティングを実施し、複数アポイント獲得。商談で得た示唆（業界特有のネットワーク制約に起因する PPAP のパスワードメール未着問題、決算時期と予算優先度の関係など）を、注力セグメントの判断材料として整理しています。

独立可能な目安額として月次売上目標を定義し、「単価×件数」の複数の組み合わせで逆算する運用に切り替えました。

##### コンテンツマーケティング ＆ プレスリリース

マーケ方針を「**SEO ブログ定期発信＋切り抜き SNS／大規模展示会出展／月1プレスリリース**」として整理し運用へ。プレスリリース配信サービスの無料枠を、展示会出展告知 → 機能リリース → コーポレートアップデートというシーケンスで計画的に消化。**プロダクト分析ツールによる行動分析**を導入し、「**CTA クリック後の CVR が低い**」など定量的な課題を発見、ブログのスクロール深度計測と CTA 飛び先 UX 改善にリソースを集中する判断を実施。

##### ターゲットセグメントの試行錯誤

ターゲット顧客を段階的にピボットしながら検証：

- 創業初期：自治体・士業を主軸候補に
- 中期：実反応を踏まえ製造業・医療にも拡張
- 直近：医療機関へのテストマーケから「小規模（〜50人）」「中堅（50〜300人）」のどちらに注力するかを、テストマーケ結果・自社の対応能力（小規模＝営業件数のスループット、中堅＝導入サポート・稟議突破資料）・キャッシュフローを軸に判断する方針を策定。

#### 3. ファイナンス・コーポレート

##### 補助金・助成金・制度活用の網羅的設計

公的な資金調達／支援制度をリサーチして実行に乗せています：

- **創業助成金**：募集要項調査を実施し、要件適合性を確認。
- **IT 導入補助金**：サービス事業者登録要件（実績要件）の調査を実施し、登録タイミングを計画。
- **公的支援機関のビジネスマッチング**：申請から訪問審査、複数年の継続支援まで完遂。
- **自治体系の媒体掲載**：出稿による接点拡大を試行。
- **資金調達フェーズの議論**：「プロダクトはできているが売上が立っていない／ようやく立ったぐらい」フェーズのプレバリュエーション相場の整理、調達手段（エクイティ／デット）と希薄化リスクの整理を経営 MTG で実施。

##### 料金プランの設計

無料／チーム／法人向けの3プラン体系を設計。営業代行パートナーからの市場フィードバック（「10〜数十人規模での年額目安」「階段状の価格」「年払い推奨」「1人刻みは避ける」）を反映し、決済プロバイダの料金 ID 構造の制約とすり合わせながら現実的な料金表に落とし込み。

### ハイライト：4人の兼業創業者で BtoB SaaS を回す仕組み

- **3か月単位のストラテジー文書 + 毎週の経営／Sprint MTG**：四半期ごとに「会社の目標／Sales／Dev／Company」を分けて立て、毎週前週の進捗をチェックし、次の意思決定をドキュメント化するサイクルを定着。経営判断・開発判断・営業判断すべてを追跡可能なドキュメントに残し、コンテキスト忘却によるムダな再議論を最小化。このドキュメントはAIによる経営戦略の考案や評価にも活用されます。
- **Coding Agent ファースト**：Coding AgentにドキュメントツールのMCP、ユーザー行動解析ツールのAPIなどを接続することで、マーケティング調査や経営戦略の考案・評価にコーディングエージェントを使い、そのアウトプットをドキュメントツールにレポートとして残すようにすることで、コーディングエージェントを組織的に活用しています。
- **セールスとプロダクト改善のループ**：展示会・テレアポ・商談から得た顧客の生の声（「鍵の登録・交換はどうするのか」「業界特有のネットワーク制約で PPAP のパスワードメールが届かない」など）を、コーディングエージェントが参照し次の施策考案に反映します。
