# 自己PR

職務経歴を貫く強みは、①エンジニアリングを土台にしたデータドリブンな意思決定、②AI を組織の集合知に変換する発想、③不確実性のオーガナイズ、④仕組みを考案し委譲するサイクル、の4つです。具体的な事例は職務経歴を参照ください。

## 1. エンジニアリングを土台にしたデータドリブンな意思決定

技術制約とユーザー価値の両方を理解した上で、グロースハック／マーケティング理論と数値分析を結びつけ、施策を発案から実行まで持っていけることが強みです。「分析できるエンジニア」「実装できる PdM」のどちらでもなく、両者を同じ思考プロセスの中で扱えることを重視しています。

意思決定の現実解は、ユーザー価値・技術制約・データから読める挙動の3つが交差する地点にあると考えており、そこに着地させる判断軸を持っていることが価値だと考えています。チーム内では分析リテラシー自体を底上げする教育にも投資しており、自分一人がボトルネックにならない構造づくりを意識しています。

## 2. AI を組織の集合知に変換する発想

Coding Agent の調教（プロンプト・コンテキスト・MCP 設定）を個人の生産性ハックに閉じさせず、リポジトリにコミットして組織で共有・進化させる対象として扱う、という思想を一貫して持っています。

AI 活用は個人スキルではなく組織能力として設計すべきだと考えており、要件定義・デザイン・コードレビューといった既存プロセスを「AI が介在することを前提にどう再設計すべきか」という観点で見直しています。シニアエンジニアの暗黙知を AI 経由で再利用可能な形に変換することにも関心があり、属人化していたレビュー知見を組織資産化する試みを継続しています。

## 3. 不確実性のオーガナイズ

研究駆動・0→1 領域のプロジェクトで「いつ・何を・どこまで出すか」のバランス調整を中心的な仕事として捉えています。学生時代からピクシブ R&D、WindHole の創業期、CoeFont の研究連携プロジェクトと、不確実性の高い領域に 7 年以上継続して身を置いてきました。

このタイプの仕事で重要なのは、研究側の成果と事業側の要請の間で「出せるのか／何を待てば出せるのか」を翻訳することと、関係者間の依存関係を可視化して合意形成のリードタイムを短くすること。ドキュメント設計（要件定義書・エンティティ別 CRUD／ライフサイクル／ビジネスロジック分割）と部門横断のコミュニケーション設計を方法論として持っています。

## 4. 仕組みを考案し委譲するサイクル

「このチームには XX が足りない」と気づいたら、制度を設計し運用に乗せ、定着したら後任に委譲する。このサイクルを回し続けることがマネージャー業務の核だと考えています。

新しい役割や運用を作って自分が背負い続けるのではなく、再現可能なフローに落とし込んで他のメンバーに渡すことを最初から設計に織り込みます。土台にあるのは、エンジニア・ステークホルダー間の橋渡しの実務経験です。技術的な専門用語を平易な例えに置き換えながら、経営層・ビジネスサイドとエンジニアの間のコミュニケーションを担ってきました。

## 補足：遠隔コミュニケーションの円滑化への興味・関心

学士・修士で Human Computer Interaction 領域の研究を行い、遠隔コミュニケーションを円滑にする目的で非言語情報が与える影響に着目したシステム発案・開発・ユーザビリティテストを実施。

[Exploring How to Display Referential Action to Support Remote Group Discussion | Proceedings of the Asian CHI Symposium 2021](https://dl.acm.org/doi/10.1145/3429360.3468188)

リモートワークにおける通信不確実性をテーマとして関心を持ち続けており、CoeFont でも「通信不確実性の低減」を Performance Goal の中核に据えるなど、実務と地続きになっています。

新世代の大規模 TTS モデルの本番リリース責任者として、2025年6月〜8月にかけてプロジェクトを統括。推論時間や対応音声キャラクターラインアップが既存モデルと異なり、新たなサービス設計・ユーザー動線設計が必要でした。

##### Search UX 改善

音声キャラクター検索画面 UX を担当 PdM として、要件定義から優先度づけまで一貫して推進。Google Analytics で検索利用状況を分析しUI改善方針を決定。検索リソースのコスト・タグの内部データモデル・関連システム連携といった技術的制約と、ユーザー価値のバランスを取りながら、プロジェクトを分割して実行のスコープ・優先順位を設定。要件・議論プロセスは GitHub に残し追跡可能な状態を整備しました。

##### サブスクリプション解約後のアップセル動線設計

Google Analytics で解約直後ユーザーの挙動を観察して仮説化し、Stripe Webhook によるサブスク状態遷移・フロントエンド状態管理・データ計測パイプラインの制約を踏まえた現実解として着地させることを得意としています。
事例：CoeFontには、プロジェクトと呼ばれる読み上げ用の専用テキストファイルがあり、有償プランのユーザーは複数個作成することができますが、無料ユーザーは1つまでの制約があります。従来は解約後は複数プロジェクトを強制削除する仕様でしたが、**解約後もサムネイルは複数見えるが編集できるのは1個のみ、残りは非活性表示でクリックするとアップセル動線へ誘導する**仕様を設計・実装に落としました。

#### 2. エンジニアリングマネジメント／スクラム運営

##### リリース運用と Repository Leader / Tech Lead 体制の議論

入社直後、フロントエンド／バックエンドのチーム分割撤廃直後で技術領域別の意思決定責任者が不在、リリース担当者の属人化が課題でした。Repository Owner（リポジトリ単位の責任）と Tech Lead（技術選定・アーキテクチャ）を分離する案を起案して議論をホスト。「正式設置は時期尚早」と判断し、まずはリリース担当者の明文化（BE/FE それぞれに固定責任者を設置）で属人化を解消する段階的アプローチを取りました。

##### インシデント対応の定義と運用整備

アドホックだったインシデント対応を、定型フローと役割分担に落とし込み再現性のある運用に変えました。

- エンドユーザー向け障害対応コミュニケーション体制の整備
- Engineering 部門全員へのインシデント対応プロセスのインストール
- CS / Sales / Marketing など各部門の対応事項定義

整備以降、自身がインシデントコマンダとして実インシデントの陣頭指揮を多数経験し、フローを実運用で磨き込みました。
ここで定めたフローはセキュリティ監査取得時に証左として提出しました。

#### 3. 組織設計・スケールの仕組みづくり

組織拡大の中で、新しい役割／フローを考案して定着させ、後任に委譲するサイクルを繰り返し回してきました。

##### タスクリーダー制度の発案

1つの PBI に FE/BE エンジニアを 2〜3人アサインする運用で、責任者不在により議論が収束しない事態が発生。PBI 単位で軽量な取りまとめ役を 1名指名する **タスクリーダー制度**を設計し、実装方針のクロージング・進行取りまとめ・レビュアー指定の責任を集約。FE/BE 間の実装方針合意を早期化しました。

##### CRE（Customer Reliability Engineering）チームの新設

CS 部署がエンジニア個人に直接対応依頼するフローで、エンジニアの実装が止まる課題に対し、Engineering 部門内に CRE を新設。CS からのチケット起票フロー、緊急度トリアージルール（インシデント疑い／要調査／要望）、通常開発と CRE 業務のリソース分離を整備。

##### Technical Talk Session（社内勉強会）の主催

組織拡大によるチーム間コミュニケーション希薄化への対応として、定期的な技術勉強会を主催。新技術紹介、他チームのプロジェクト紹介、ベストプラクティス共有のほか、新人エンジニアの発表機会を提供。AWS Builders Online Series など外部技術プログラムへの参加もエンジニア／マーケティング部門に展開。

#### 4. AI コーディングエージェントを前提とした開発プロセス再設計

2025年以降、Claude Code が実用に乗ったタイミングで開発プロセス全体を AI 前提に再設計することを主導。基本思想は、Coding Agent の調教（プロンプト・コンテキスト・MCP接続）を個人最適化に閉じさせず、リポジトリにコミットして組織の集合知として育てていくこと。

- **要件定義プロセスの GitHub 移行**：PRD・要件・PBI を Markdown で GitHub 管理。Notion で散在していた要件定義を Diff / Review / 履歴管理可能な状態にし、PR ベースで PRD レビューを受けるフローへ転換。
- **Claude Code への調査基盤接続**：調査系 MCP・社内 API を Claude Code に接続し、要件作成の調査・背景情報補足・PRD 妥当性評価を Agentic に回せる基盤を整備。
- **エージェント定義のリポジトリ化**：各メンバーのエージェント定義（プロンプト・カスタムコマンド・MCP 設定）をリポジトリにコミットする運用に統一。
- **レビュー知見の AI プロンプト化基盤**：GitHub PR のレビューコメントを diff hunk・ファイルパス・行番号・コミット情報とともに構造化 JSON で抽出する CLI ツール [pr-review-data-collector](https://github.com/krgpi/pr-review-data-collector) を自作。シニアエンジニアの過去レビュー指摘を AI コードレビュー Bot 用の学習・プロンプトデータセットとして再利用できる形に整え、暗黙知を組織横断で活用する仕組みを試作。
